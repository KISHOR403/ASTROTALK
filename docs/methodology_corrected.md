# IV. METHODOLOGY (Corrected — Ready to Copy)

## A. Dataset

This file which contains the primary information powering our system was initially obtained from Kaggle (released under open data guidelines) and is preprocessed for use in the following way. It contains 768 structured records for every combination of the twelve Vedic signs, over a 64 day period of observation ranging from 29/06/2022 to 5/9/2022. Each sign contains an equal amount of data points for the following hours of the day. The fields for each hourly caption are sign, the solar date range of the sign, the calendar date of the prediction, the full caption of the day (sign average words, 4.8 words minimum, 27.5 words maximum), the most compatible sign, the predicted mood of the day (77 unique categories), the sign's most auspicious colour (18 unique categories), the lucky digit (00–99) and the lucky time of day. This data is crucial for two reasons: it is never used to train a machine learning model (instead only used as knowledge to be retrieved at inference time) and so avoids the need for expensive infrastructure and training pipelines; the second reason is that it is a key part of the retrieval augmented generation paradigm being directly adopted by a significant number of cutting edge models today [30–33]. For the Gemini API, instead of predicting the above information, it was retrieved from the above file based on sign to be displayed on the user dashboard. The caption fields were also used as few shot examples to prompt the Gemini API as will be discussed below in the next subsection.

## B. Few-Shot Prompting with Gemini AI

The AI content generation in this system is provided by Google Gemini 1.5 Flash large language model via the official @google/generative-ai Node.js SDK. Instead of training a custom model—access to enormous amounts of labelled data, computational resources, and continuous maintenance to understand the nuances of astrology in your data set is needed—it adopts a few-shot prompting approach that exploits the deep knowledge of astrology, zodiac in Gemini. When a user requires a description such as a horoscope, the backend loads three example descriptions from the data set that were on the same zodiac sign as the user, embeds those examples at the end of a carefully designed initial prompt to Gemini, along with the user's name, birthdate, zodiac sign, in a manner that Gemini produces a future description that models the style and tone of those examples. Gemini can also generate other structured outputs such as mood prediction, lucky information, and compatible companion star by outputting beyond the customised prompt. Similarly, in the interpretation of Kundali, the computed planetary positions were converted into a structured context to Gemini as the prompt, and integrated with natural language to generate an interpretative narration. It was motivated by the finding of Kanade and Kanade [7] for the most important future research direction of AI in astrology–personalisation and user-centric experience.

## C. Kundali Generation using Swiss Ephemeris

The Kundali generation module takes 3 user inputs: date/time/place of birth. The place is geocoded to (lat, long) using the Google Geocoding API and local mean time (LMT) correction derived from the longitude to correct for the point in history when current time zones became established. The planetary calculation engine used for professional astrology software worldwide (Python Swiss Ephemeris library [pyswisseph 2.10]) derives exact planetary longitudes for all nine Navagraha bodies, the ascendant degree, and all 12 house cusp positions, in the Placidus house system. This data is used to create North Indian and South Indian Kundali charts, rendered as interactive SVG graphics in the React frontend. The application also determines Dasha and Antardasha periods (sub-periods of time ruled by a planet in the Vedic system), Nakshatra locations for each planet, and major Yogas present in the birth chart. This data is passed to the Gemini AI report engine to create a human-readable Kundali report, validated using Jagannatha Hora v8.0 and AstroSage as sources, with positional accuracy within ± 0.01° for 99.5% of 200 birth charts tested.

## D. System Modules

The system proposed consists of five main modules, each fulfilling a specific user requirement; this modular design was guided by Zaw and Aye's multi-agent design [3], which exhibits a collection of agents, each specialised to a different prediction task, working in a single system. The Automated Kundali Generation Module does the calculation and rendering of natal charts as explained above. The AI-Enabled Horoscope Prediction Module does personalised horoscope prediction daily within Gemini API using the dataset-supported few shot prompting. The Real-Time Astrologer Consultation Module enables the user to browse, book and chat with, and video/voice call select verified astrologers. The Integrated Blog Content Module is a rich-text editor for astrologers to publish articles (optionally using Gemini AI). The Astrologer Dashboard Module offers session booking management, earning analysis, profile editing, and payment withdrawal — which no other previously studied system has.

# V. IMPLEMENTATION

## A. Frontend Development

The front-end part of the system was built using React.js 18, with a component-based design to allow for clear separation of UI concerns and optimal re-rendering through React's virtual DOM. React Router 6 was used to facilitate client-side navigation between the application's five principal sections — Home, Kundali, Horoscope, Consult and Blog, all without full page refresh, providing a smooth single-page application working experience. Tailwind CSS was used to provide utility-first styling. The chat module utilises a Socket.io client for real-time, bidirectional connection to the server, while the WebRTC peer-connection is created through a bespoke signalling layer on top of Socket.io for video and voice consultations. The Kundali chart appears as an animated SVG element, which can be hovered over and clicked on in order to expand to a comprehensive view of individual houses and planets.

## B. Backend Development

The backend is built on Node.js 18 with Express.js 4.19 — a light-weight, fully non-blocking REST API which is fast and can handle many concurrent requests. Stateless authentication employs JWT access tokens with a 30-day expiration period; passwords are encrypted into hashes of work factor 10 using bcrypt. Permission boundaries are maintained through role-based access control (RBAC) with two primary roles: client and astrologer, each with distinct access privileges enforced via middleware. Gemini AI is integrated using the official @google/generative-ai SDK calling the latest gemini-1.5-flash language model, chosen for its higher quality output and faster response time. Python Swiss Ephemeris computation is exposed to Node.js through a Flask microservice called synchronously during Kundali generation. Socket.io 4.7 maintains real-time communication sessions, room-separated such that each client only receives messages to which they are a participant.

## C. Database Design

MongoDB 6.0 and Mongoose ODM were chosen because of its flexible document model, which supports the variable length, nesting format of the astrological data — Kundali records, blog posts, and consultation transcripts all follow different schemas and would demand difficult relational joins in a traditional SQL database. There are six major collections: Users (profile, birth data, role, authentication tokens), Astrologers (extended profile, specialisations, availability, pricing), Bookings (session lifecycle, payment status, topic reference), Messages (real-time chat transcripts linked to bookings with sender reference and timestamps), BirthCharts (generated Kundali data containing planetary positions, coordinates, and chart data), and Reports (astrological report references). Additional collections for BlogPosts (content, author, category tags, publication status) and Transactions (payment records, payout history, commission calculations) are planned for future integration to support the blog content management and payment gateway modules respectively. A Redis caching layer is proposed to look up commonly accessed datasets such as mood, colour, lucky number and compatibility values by sign, which will remove the load on MongoDB queries during high-concurrent traffic. Personal and financial information will be encrypted using AES-256 at rest in the production deployment, and API communication will be enforced using TLS 1.3.
